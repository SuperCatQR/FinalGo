## 题型与答题模板

> 本区块 ⭐✋——以下模板为原创复习方法论，不引用任何第三方真题原文；PR 审查后可结合已授权真题样本调整题型分布。

### 题型一：概念辨析类（单项选择 / 填空 / 简答）

**（1）题型特征**

题目通常要求识别数据库术语、填写概念名称，或比较两个易混概念。例如数据库与数据库系统、候选码与主码、WHERE 与 HAVING、存储过程与存储函数。

**（2）答题结构框架**

1. 先定位术语所属层级：数据对象、系统结构、关系模型、SQL 语句、事务安全还是数据管理发展。
2. 写出核心定义，避免把组成部分和作用混为一谈。
3. 对比题要列“共同点 + 差异点 + 典型使用场景”。
4. 若是选择题，用关键词排除：例如“分组后过滤”对应 HAVING，“约束外键引用”对应参照完整性。

**（3）评分要点**

| 维度 | 得分要点 | 常见扣分项 |
| --- | --- | --- |
| 定义准确 | 术语名称、作用对象和功能边界正确 | 把 DBMS 写成数据库，或把 DBS 写成单一软件 |
| 区分清晰 | 能说明相近概念差异 | 只背定义，不说明候选码与主码关系 |
| 场景匹配 | 能用例子说明使用位置 | 把 WHERE 用于聚合结果过滤 |

**（4）完整示例**

> **题目概要**：说明数据库系统三级模式结构如何支持数据独立性。
>
> **答题过程**：先写出外模式、模式、内模式三个层次；再说明外模式 / 模式映像支持逻辑独立性，模式 / 内模式映像支持物理独立性；最后说明应用程序通过外模式访问数据，底层存储变化不必直接影响用户视图。
>
> **最终答案**：三级模式结构把用户视图、全局逻辑结构和物理存储结构分离，并通过两层映像维护层次之间的对应关系。逻辑结构调整时，只要外模式 / 模式映像保持用户视图稳定，应用程序可少受影响；物理存储改变时，只要模式 / 内模式映像维护逻辑结构不变，也不影响上层逻辑访问。因此三级模式结构提高了逻辑独立性和物理独立性。

### 题型二：SQL 读写操作类

**（1）题型特征**

题目给出若干关系表和查询 / 更新需求，要求写 SQL 或判断 SQL 执行结果。常见考点包括建表、插入、更新、删除、单表查询、多表连接、分组聚合、排序、分页和视图。

**（2）答题结构框架**

1. 把自然语言需求拆成目标列、数据来源、连接条件、过滤条件、分组条件、分组后过滤、排序 / 限制。
2. 写 SELECT 框架：`SELECT` → `FROM` → `JOIN/ON` → `WHERE` → `GROUP BY` → `HAVING` → `ORDER BY` → `LIMIT`。
3. 对增删改题先确认作用表和条件，再写 `INSERT` / `UPDATE` / `DELETE`；避免无条件更新或删除。
4. 对 DDL 题先列字段、数据类型、主键 / 外键 / 约束，再写建表或索引语句。

**（3）评分要点**

| 维度 | 得分要点 | 常见扣分项 |
| --- | --- | --- |
| 语句结构 | SQL 子句顺序和关键字正确 | HAVING 写在 GROUP BY 前，或漏 FROM |
| 条件表达 | 连接条件、过滤条件与题意一致 | 多表查询漏连接条件导致笛卡尔积 |
| 聚合处理 | 分组字段、聚合函数、HAVING 使用正确 | 在 WHERE 中过滤聚合结果 |
| 数据安全 | 更新 / 删除有明确 WHERE 条件 | 无条件 UPDATE/DELETE |

**（4）完整示例**

> **题目概要**：某系统有学生表和成绩表，要求统计每个学生的平均成绩，只显示平均成绩不低于 70 分的学生，并按平均成绩降序排列。
>
> **答题过程**：目标列是学生标识与平均成绩；来源是学生表连接成绩表；按学生分组；聚合结果过滤用 HAVING；最后降序排序。
>
> **最终答案**：可写为 `SELECT s.student_id, s.name, AVG(g.score) AS avg_score FROM students s JOIN grades g ON s.student_id = g.student_id GROUP BY s.student_id, s.name HAVING AVG(g.score) >= 70 ORDER BY avg_score DESC;`。实际字段名以题目给定表结构为准。

### 题型三：E-R 建模与关系模式转换类

**（1）题型特征**

题目以业务描述给出若干对象、属性和业务规则，要求画局部 / 全局 E-R 图，或把 E-R 图转换为关系模式并标明主键、外键。

**（2）答题结构框架**

1. 找实体：业务中可独立管理的名词，如学生、课程、订单、仓库。
2. 找属性：描述实体的字段，区分普通属性、标识属性和可能的派生属性。
3. 找联系：判断实体之间是 1:1、1:N 还是 M:N，并识别联系属性。
4. 转关系模式：实体转表；1:N 通常在 N 端加入外键；M:N 通常转为新关系并包含双方主键和联系属性。
5. 标注主键、外键与必要完整性约束。

**（3）评分要点**

| 维度 | 得分要点 | 常见扣分项 |
| --- | --- | --- |
| 实体抽取 | 实体与属性边界合理 | 把属性误建成实体，或遗漏核心实体 |
| 联系基数 | 1:1、1:N、M:N 判断正确 | 多对多未单独转关系 |
| 模式转换 | 主键、外键和联系属性完整 | 关系模式缺主键或外键方向错误 |
| 约束说明 | 能体现业务规则 | 未说明唯一性、非空或参照约束 |

**（4）完整示例**

> **题目概要**：图书借阅系统管理读者、图书和借阅记录。一个读者可借多本图书，一本图书也可被不同读者在不同时间借阅。
>
> **答题过程**：读者和图书是实体；借阅是读者与图书之间的多对多联系，并带借出日期、归还日期等联系属性；转换时应新增借阅关系。
>
> **最终答案**：可设计 `Reader(reader_id, name, ...)`、`Book(book_id, title, ...)`、`Borrow(reader_id, book_id, borrow_date, return_date, ...)`。`Borrow` 的主键可按业务设为 `(reader_id, book_id, borrow_date)` 或单独借阅编号，`reader_id` 和 `book_id` 分别是外键。

### 题型四：函数依赖与规范化分析类

**（1）题型特征**

题目给出一个关系模式及函数依赖集，要求求候选码、判断范式、说明异常，或分解到 2NF / 3NF / BCNF。

**（2）答题结构框架**

1. 根据函数依赖求属性闭包，找出能决定全部属性的候选码。
2. 区分主属性和非主属性。
3. 判断是否满足 1NF；若存在非主属性对候选码的部分依赖，则不满足 2NF。
4. 若存在非主属性对候选码的传递依赖，则不满足 3NF。
5. 若每个非平凡函数依赖的决定因素都是超码，则满足 BCNF；否则需继续分解。
6. 分解时说明目标：减少冗余、消除更新 / 插入 / 删除异常，并尽量保持无损连接与依赖保持。

**（3）评分要点**

| 维度 | 得分要点 | 常见扣分项 |
| --- | --- | --- |
| 候选码 | 属性闭包推导正确 | 漏掉组合码或误把非码当主码 |
| 依赖类型 | 能区分部分依赖和传递依赖 | 只写“有冗余”不指出依赖原因 |
| 范式判断 | 判断顺序和理由完整 | 跳过 2NF / 3NF 直接判 BCNF |
| 分解结果 | 分解后关系含义清楚 | 分解导致信息丢失或主键不明 |

**（4）完整示例**

> **题目概要**：某成绩关系包含学生、课程、教师和成绩信息，其中学生与课程共同决定成绩，课程决定任课教师。
>
> **答题过程**：候选码通常是“学生 + 课程”；“课程 → 教师”使非主属性教师依赖于候选码的一部分或经由课程产生冗余；应把课程与教师信息拆出。
>
> **最终答案**：可分解为 `Score(student_id, course_id, score)` 与 `Course(course_id, teacher_id, ...)`。这样成绩事实与课程任课信息分离，减少同一课程教师信息重复存储造成的更新异常。

### 题型五：事务、安全与恢复场景分析类

**（1）题型特征**

题目给出并发访问、数据损坏、权限控制或备份恢复场景，要求说明问题类型、原因和控制手段。常见考点包括 ACID、封锁、死锁、两段封锁、权限授予 / 撤销、触发器、备份恢复。

**（2）答题结构框架**

1. 识别场景：并发异常、完整性破坏、权限越权、故障恢复还是审计自动化。
2. 对事务题写 ACID 中被威胁的性质，并说明并发控制或日志恢复手段。
3. 对权限题说明用户、对象、操作权限和授权 / 撤销语义。
4. 对触发器题说明触发事件、触发时机、触发条件和动作。
5. 对备份恢复题说明备份方式、恢复命令和恢复点。

**（3）评分要点**

| 维度 | 得分要点 | 常见扣分项 |
| --- | --- | --- |
| 问题识别 | 正确判断脏读、不可重复读、丢失修改、死锁等 | 把所有并发问题笼统写成“数据不一致” |
| 控制机制 | 能匹配锁、事务隔离、权限、触发器、备份恢复 | 只写概念，不说明适用场景 |
| 结果说明 | 能解释机制如何保护一致性、安全性或持久性 | 没有说明代价，如锁粒度影响并发度 |

**（4）完整示例**

> **题目概要**：两个事务同时修改同一账户余额，后提交的事务覆盖了先提交事务的修改。
>
> **答题过程**：这是并发控制中的丢失修改问题，影响一致性和隔离性；应通过排他锁、合适隔离级别或两段封锁协议控制并发写。
>
> **最终答案**：数据库应在事务修改共享数据前获取排他锁，在事务完成前保持必要锁，避免另一个事务在未协调的情况下读取或覆盖同一数据项。若系统采用两段封锁协议，可在增长阶段获取所需锁、收缩阶段释放锁，从而提高冲突可串行化的保证，但也需注意死锁检测或预防。

### 典型题解析思路（空态保留）

#### 典型题 1：SQL 多表查询与聚合统计（待统计重复考次）

- **考查方式**：通常给出两到三张表，要求按业务条件查询、统计或排序。
- **通用解题路径**：
  1. 画出表之间的主外键关系，先确定连接条件。
  2. 把题目中的“每个 / 每类 / 按……统计”转换为 `GROUP BY`。
  3. 把“统计后满足……”转换为 `HAVING`，把普通行条件保留在 `WHERE`。
  4. 最后处理排序、别名与限制行数。
- **常见变体**：多表连接、嵌套查询、聚合函数、分组后过滤、视图查询。

#### 典型题 2：E-R 设计与关系模式转换（待统计重复考次）

- **考查方式**：给出业务文字描述，要求画 E-R 图或写关系模式。
- **通用解题路径**：
  1. 标记业务名词，区分实体、属性和联系。
  2. 判断联系基数：一对一、一对多、多对多。
  3. 多对多联系单独转关系，一对多通常把“一”端主键放到“多”端。
  4. 在关系模式中标明主键、外键和必要约束。
- **常见变体**：联系带属性、弱实体、局部 E-R 图合并、冗余联系识别。

#### 典型题 3：函数依赖与范式分解（待统计重复考次）

- **考查方式**：给出关系模式和函数依赖，要求判断范式或分解关系。
- **通用解题路径**：
  1. 求候选码：用属性闭包判断哪些属性组能推出全部属性。
  2. 找非主属性，检查是否存在部分依赖或传递依赖。
  3. 根据 2NF、3NF、BCNF 条件逐层判断。
  4. 分解时保留业务语义，并说明消除的冗余或异常。
- **常见变体**：候选码不唯一、依赖集中存在传递链、BCNF 与 3NF 判断差异。

#### 典型题 4：事务与并发控制（待统计重复考次）

- **考查方式**：给出两个或多个事务交错执行的场景，要求识别异常或选择控制方法。
- **通用解题路径**：
  1. 识别读写顺序，判断是否出现丢失修改、脏读、不可重复读或死锁。
  2. 写出受影响的 ACID 性质。
  3. 选择锁、隔离级别、两段封锁或恢复机制。
  4. 说明控制机制带来的并发度代价。
- **常见变体**：共享锁 / 排他锁、锁粒度、死锁检测、备份恢复。
